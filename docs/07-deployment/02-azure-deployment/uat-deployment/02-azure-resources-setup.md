# STEP-02: Azure Resources Setup（App 層資源建立）

> **目的**：建立 UAT 環境 App 層 6 大資源並完成 Managed Identity + RBAC 配置。
>
> **重要原則**：本 STEP 為 mutation 操作（會建立 Azure 資源並產生費用），`requires_approval: true`，AI 助手必須在執行前向 infra-admin 取得確認。

---

## 🎯 1. Objective

建立 UAT 環境的 App 層 Azure 資源（不含 Infra Team 負責的基礎架構）：

| 資源 | 用途 | SKU（UAT）|
|------|------|----------|
| Azure Container Registry（ACR）| Docker image 倉庫 | Standard |
| Azure Container Apps Environment | Container App 執行環境 | Consumption |
| PostgreSQL Flexible Server | 主資料庫 | Burstable B2s（PG 15）|
| Storage Account + Blob Container | 文件儲存（`documents`）| Standard_LRS |
| Application Insights | 監控與遙測 | 連 Infra Team 提供的 Log Analytics |
| Managed Identity（System-Assigned）| 取代 secret 認證 | 自動建立 |

**Mode C 邊界提醒**：
- Resource Group `${RG_NAME}` **必須**由 Infra Team 預先建立（STEP-01 已驗證）
- Key Vault 採二選一（共享或自建），本 STEP **不建立** Key Vault（由 STEP-03 處理）
- VNet / Subnets 由 Infra Team 提供（若採 Managed VNet 則 Container Apps 自行管理）

## 🔧 Action 2.1: Verify Resource Group Ready（前置條件複查）

> **重要**：本 Action 只做 **read-only** 檢查，是 STEP-01 Action 1.4 的快速複查（避免 RG 在 STEP-01 → STEP-02 之間被刪除/變更）。

### Command
```bash
# 確認 RG 存在且 provisioningState=Succeeded
az group show \
  --name "${RG_NAME}" \
  --output json
```

### Verify
```bash
# 取得 provisioningState
az group show --name "${RG_NAME}" --query "properties.provisioningState" -o tsv

# 確認當前 user 對 RG 仍有 Contributor 角色
az role assignment list \
  --assignee "$(az account show --query "user.name" -o tsv)" \
  --scope "/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RG_NAME}" \
  --query "[?roleDefinitionName=='Contributor' || roleDefinitionName=='Owner'].roleDefinitionName" \
  --output tsv

# 確認 Location 與 ${LOCATION} 一致
az group show --name "${RG_NAME}" --query "location" -o tsv
```

### Expected Output
```
# provisioningState
Succeeded

# Role assignment
Contributor

# Location（與 STEP-00 §5 ${LOCATION} 一致）
southeastasia
```

### If Fails
- Error: `ResourceGroupNotFound` → RG 被誤刪，**立即停止**並通知 Infra Team 重建
- Error: 無 Contributor 角色 → 聯繫 infra-admin 重新指派 RBAC（不可繼續）
- Location 不符 → 確認 STEP-00 §5 環境變數，避免跨 region 部署

## 🔧 Action 2.4: Create PostgreSQL Flexible Server

> **設計重點**：UAT 用 Burstable B2s（成本優先），PostgreSQL 15（與本地一致），Backup 7 天，**先使用 admin password，STEP-03 後改 Managed Identity 認證**。

### Command
```bash
# 從 KeyVault / 安全來源取得 admin password（不要硬編碼）
# 此 password 將在 STEP-03 寫入 Key Vault 後從 source code / shell history 清除
read -s -p "Enter PostgreSQL admin password (min 12 chars): " PG_ADMIN_PASSWORD
echo

# 建立 PostgreSQL Flexible Server（B2s Burstable, PG 15, 7-day backup）
az postgres flexible-server create \
  --resource-group "${RG_NAME}" \
  --name "${POSTGRES_NAME}" \
  --location "${LOCATION}" \
  --tier Burstable \
  --sku-name Standard_B2s \
  --version 15 \
  --storage-size 32 \
  --storage-auto-grow Enabled \
  --backup-retention 7 \
  --geo-redundant-backup Disabled \
  --admin-user pgadmin \
  --admin-password "${PG_ADMIN_PASSWORD}" \
  --high-availability Disabled \
  --tags environment=uat owner=app-team change=CHANGE-055 \
  --yes \
  --output json

# 立即清除 shell 變數（防止意外寫入 log）
unset PG_ADMIN_PASSWORD
```

### Verify
```bash
# Polling 直到 state=Ready（典型 5-15 分鐘）
while true; do
  STATE=$(az postgres flexible-server show \
    --resource-group "${RG_NAME}" \
    --name "${POSTGRES_NAME}" \
    --query "state" -o tsv 2>/dev/null)
  echo "$(date +%H:%M:%S) PostgreSQL state: ${STATE}"
  [[ "${STATE}" == "Ready" ]] && break
  [[ "${STATE}" == "Failed" ]] && { echo "Provisioning FAILED"; exit 1; }
  sleep 60
done

# 取得 FQDN 與配置摘要
az postgres flexible-server show \
  --resource-group "${RG_NAME}" \
  --name "${POSTGRES_NAME}" \
  --query "{fqdn: fullyQualifiedDomainName, version: version, sku: sku.name, tier: sku.tier, state: state, backupRetention: backup.backupRetentionDays}" \
  --output json
```

### Expected Output
```json
{
  "fqdn": "psql-aidocextract-uat.postgres.database.azure.com",
  "version": "15",
  "sku": "Standard_B2s",
  "tier": "Burstable",
  "state": "Ready",
  "backupRetention": 7
}
```

### If Fails
- Error: `ServerNameAlreadyExists` → DNS 名稱衝突，調整 `${POSTGRES_NAME}` 後重試
- Error: `QuotaExceeded` → 提交 PostgreSQL vCPU quota request
- Error: `PasswordTooSimple` → 密碼需 ≥12 字元，含大小寫 + 數字 + 符號
- 超過 20 分鐘仍 `Provisioning` → 不要刪除重建，繼續 poll 或聯繫 Azure Support
- ⚠️ **預估 duration**: 5-15 分鐘（最長至 20 分鐘）
- ⚠️ **絕對禁止**：不可使用 `--yes` 跳過互動式 prompt 但又把密碼寫入 log

### 📝 Bicep 替代路徑
```bash
# 注意：Bicep 無法安全傳 password，建議用 az CLI 建好後再 import 到 Bicep state
az deployment group create \
  --resource-group "${RG_NAME}" \
  --template-file "infrastructure/bicep/modules/postgres.bicep" \
  --parameters postgresName="${POSTGRES_NAME}" location="${LOCATION}" \
               adminLogin=pgadmin tier=Burstable skuName=Standard_B2s \

```

## 🔧 Action 2.5: Create Storage Account + Blob Container

> **設計重點**：UAT 用 LRS（成本優先），啟用 Soft-delete（7 天，誤刪保護），Container 命名 `documents`（與 `AZURE_STORAGE_CONTAINER` env 一致）。

### Command
```bash
# 建立 Storage Account（Standard_LRS, soft-delete 啟用）
az storage account create \
  --resource-group "${RG_NAME}" \
  --name "${STORAGE_NAME}" \
  --location "${LOCATION}" \
  --sku Standard_LRS \
  --kind StorageV2 \
  --access-tier Hot \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false \
  --allow-shared-key-access true \
  --tags environment=uat owner=app-team change=CHANGE-055 \
  --output json

# 啟用 Blob soft-delete（7 天）+ Container soft-delete（7 天）
az storage account blob-service-properties update \
  --resource-group "${RG_NAME}" \
  --account-name "${STORAGE_NAME}" \
  --enable-delete-retention true \
  --delete-retention-days 7 \
  --enable-container-delete-retention true \
  --container-delete-retention-days 7 \
  --enable-versioning true \
  --output json

# 建立 documents container（透過 OAuth/AAD 認證，避免取 account key）
az storage container create \
  --account-name "${STORAGE_NAME}" \
  --name documents \
  --auth-mode login \
  --public-access off \
  --output json
```

### Verify
```bash
# 確認 Storage Account 配置
az storage account show \
  --resource-group "${RG_NAME}" \
  --name "${STORAGE_NAME}" \
  --query "{name: name, sku: sku.name, kind: kind, tlsVersion: minimumTlsVersion, publicBlobAccess: allowBlobPublicAccess, status: provisioningState}" \
  --output json

# 確認 soft-delete 已啟用
az storage account blob-service-properties show \
  --resource-group "${RG_NAME}" \
  --account-name "${STORAGE_NAME}" \
  --query "{blobSoftDelete: deleteRetentionPolicy.enabled, blobRetentionDays: deleteRetentionPolicy.days, containerSoftDelete: containerDeleteRetentionPolicy.enabled, versioning: isVersioningEnabled}" \
  --output json

# 確認 documents container 存在
az storage container show \
  --account-name "${STORAGE_NAME}" \
  --name documents \
  --auth-mode login \
  --query "{name: name, publicAccess: properties.publicAccess}" \
  --output json
```

### Expected Output
```json
{
  "name": "staidocextractuat",
  "sku": "Standard_LRS",
  "kind": "StorageV2",
  "tlsVersion": "TLS1_2",
  "publicBlobAccess": false,
  "status": "Succeeded"
}
{
  "blobSoftDelete": true,
  "blobRetentionDays": 7,
  "containerSoftDelete": true,
  "versioning": true
}
{
  "name": "documents",
  "publicAccess": null
}
```

### If Fails
- Error: `StorageAccountAlreadyTaken` → DNS 名稱衝突，調整 `${STORAGE_NAME}`（必須全 lowercase + ≤24 字）
- Error: `ContainerNameInvalid` → Container 名只能含 `[a-z0-9-]`，不可大寫
- Error: `AuthorizationPermissionMismatch` → 當前 user 缺 `Storage Blob Data Contributor`，臨時用 `--auth-mode key` 但完成後 STEP-07 必改 MI
- ⚠️ **絕對禁止**：`--allow-blob-public-access true` 或 `--public-access blob/container`

### 📝 Bicep 替代路徑
```bash
az deployment group create \
  --resource-group "${RG_NAME}" \
  --template-file "infrastructure/bicep/modules/storage.bicep" \
  --parameters storageName="${STORAGE_NAME}" location="${LOCATION}" \
               sku=Standard_LRS containerName=documents
```

## 🔧 Action 2.7: Configure System-Assigned Managed Identity + RBAC

> **設計重點**：MI 在 STEP-08 Container App 建立時才會出現，本 Action 預先**不建立 Container App**（留待 STEP-08），但**預先計算 RBAC 賦權清單**並產出腳本。實際 RBAC 賦權延遲到 STEP-08 後執行。
>
> **替代方案**：若採用 User-Assigned MI（pre-create），此 Action 可立即執行；本 STEP 採 System-Assigned，故僅產出檢查清單。

### Command
```bash
# 由於 System-Assigned MI 隨 Container App 建立，本 Action 產出 RBAC 賦權「待執行清單」
# 寫入暫存檔，待 STEP-08 完成 Container App 後執行

cat > "${PROJECT_ROOT}/deployment-state/rbac-pending.sh" <<'BASH_EOF'
#!/usr/bin/env bash
# RBAC 賦權待執行腳本（在 STEP-08 完成後執行）
# 取得 Container App 的 System-Assigned MI principal ID

set -euo pipefail

CA_PRINCIPAL_ID=$(az containerapp show \
  --resource-group "${RG_NAME}" \
  --name "${CA_NAME}" \
  --query "identity.principalId" -o tsv)

echo "Container App MI Principal ID: ${CA_PRINCIPAL_ID}"

# === RBAC 賦權清單 ===

# 1. Key Vault Secrets User（讀 secret）
KV_RESOURCE_ID=$(az keyvault show --name "${KV_NAME}" --query id -o tsv)
az role assignment create \
  --role "Key Vault Secrets User" \
  --assignee-object-id "${CA_PRINCIPAL_ID}" \
  --assignee-principal-type ServicePrincipal \
  --scope "${KV_RESOURCE_ID}"

# 2. Storage Blob Data Contributor（讀寫 documents container）
STORAGE_RESOURCE_ID=$(az storage account show \
  --resource-group "${RG_NAME}" --name "${STORAGE_NAME}" --query id -o tsv)
az role assignment create \
  --role "Storage Blob Data Contributor" \
  --assignee-object-id "${CA_PRINCIPAL_ID}" \
  --assignee-principal-type ServicePrincipal \
  --scope "${STORAGE_RESOURCE_ID}"

# 3. AcrPull（從 ACR pull image）
ACR_RESOURCE_ID=$(az acr show --name "${ACR_NAME}" --query id -o tsv)
az role assignment create \
  --role "AcrPull" \
  --assignee-object-id "${CA_PRINCIPAL_ID}" \
  --assignee-principal-type ServicePrincipal \
  --scope "${ACR_RESOURCE_ID}"

# 4. Monitoring Metrics Publisher（寫 App Insights）— 通常 connection string 已足夠
# 視 Phase 2 觀察決定是否需要

echo "RBAC 賦權完成（3 個角色）"
BASH_EOF

chmod +x "${PROJECT_ROOT}/deployment-state/rbac-pending.sh"
echo "OK: RBAC 賦權腳本已產出於 deployment-state/rbac-pending.sh（STEP-08 後執行）"
```

### Verify
```bash
# 確認腳本已產出
test -x "${PROJECT_ROOT}/deployment-state/rbac-pending.sh" && echo "OK" || echo "MISSING"

# 確認三個目標資源 ID 可被解析（pre-flight check）
az keyvault show --name "${KV_NAME}" --query id -o tsv 2>/dev/null || echo "Note: Key Vault 將在 STEP-03 建立"
az storage account show --resource-group "${RG_NAME}" --name "${STORAGE_NAME}" --query id -o tsv
az acr show --name "${ACR_NAME}" --query id -o tsv
```

### Expected Output
```
OK
# Storage 與 ACR 應有 Resource ID 輸出
/subscriptions/.../resourceGroups/.../providers/Microsoft.Storage/storageAccounts/staidocextractuat
/subscriptions/.../resourceGroups/.../providers/Microsoft.ContainerRegistry/registries/acraidocextractuat
```

### If Fails
- 腳本產出失敗 → 確認 `deployment-state/` 目錄存在且有寫入權限
- KV 不存在 → 預期行為（STEP-03 才建），腳本中已用 KV_NAME 預先參數化
- ⚠️ **絕對禁止**：在本 STEP 執行 `az role assignment create`（MI 還未建立）

### 📝 Bicep 替代路徑
Bicep 中 RBAC 可用 `Microsoft.Authorization/roleAssignments` resource 在 Container App module 內 inline 處理。詳見 `infrastructure/bicep/modules/container-app.bicep` 的 `roleAssignments` 區段。

## 🔧 Action 2.8: Save Outputs to deployment-state/uat.yaml

> **設計重點**：用 `yq` 寫入而非 `sed`（YAML-aware，避免破壞結構）；只寫入**非敏感**資源 metadata，敏感資料（connection string）改寫入 Key Vault（STEP-03）。

### Command
```bash
STATE_FILE="${PROJECT_ROOT}/deployment-state/uat.yaml"
NOW=$(date -u +%Y-%m-%dT%H:%M:%SZ)

# 取得各資源 outputs
ACR_LOGIN_SERVER=$(az acr show --name "${ACR_NAME}" --query "loginServer" -o tsv)
CAE_DEFAULT_DOMAIN=$(az containerapp env show --resource-group "${RG_NAME}" --name "${CAE_NAME}" --query "properties.defaultDomain" -o tsv)
CAE_RESOURCE_ID=$(az containerapp env show --resource-group "${RG_NAME}" --name "${CAE_NAME}" --query "id" -o tsv)
PG_FQDN=$(az postgres flexible-server show --resource-group "${RG_NAME}" --name "${POSTGRES_NAME}" --query "fullyQualifiedDomainName" -o tsv)
STORAGE_BLOB_ENDPOINT=$(az storage account show --resource-group "${RG_NAME}" --name "${STORAGE_NAME}" --query "primaryEndpoints.blob" -o tsv)
APPI_RESOURCE_ID=$(az monitor app-insights component show --resource-group "${RG_NAME}" --app "${APP_INSIGHTS_NAME}" --query "id" -o tsv)

# 用 yq 原子性寫入（不寫入 secrets / connection strings）
yq -i ".metadata.last_updated = \"${NOW}\"" "${STATE_FILE}"

yq -i ".resources.acr.name = \"${ACR_NAME}\"" "${STATE_FILE}"
yq -i ".resources.acr.login_server = \"${ACR_LOGIN_SERVER}\"" "${STATE_FILE}"
yq -i ".resources.acr.sku = \"Standard\"" "${STATE_FILE}"
yq -i ".resources.acr.admin_enabled = false" "${STATE_FILE}"

yq -i ".resources.container_apps_env.name = \"${CAE_NAME}\"" "${STATE_FILE}"
yq -i ".resources.container_apps_env.id = \"${CAE_RESOURCE_ID}\"" "${STATE_FILE}"
yq -i ".resources.container_apps_env.default_domain = \"${CAE_DEFAULT_DOMAIN}\"" "${STATE_FILE}"

yq -i ".resources.postgres.name = \"${POSTGRES_NAME}\"" "${STATE_FILE}"
yq -i ".resources.postgres.fqdn = \"${PG_FQDN}\"" "${STATE_FILE}"
yq -i ".resources.postgres.version = \"15\"" "${STATE_FILE}"
yq -i ".resources.postgres.tier = \"Burstable\"" "${STATE_FILE}"
yq -i ".resources.postgres.sku = \"Standard_B2s\"" "${STATE_FILE}"
yq -i ".resources.postgres.backup_retention_days = 7" "${STATE_FILE}"

yq -i ".resources.storage.name = \"${STORAGE_NAME}\"" "${STATE_FILE}"
yq -i ".resources.storage.blob_endpoint = \"${STORAGE_BLOB_ENDPOINT}\"" "${STATE_FILE}"
yq -i ".resources.storage.container_name = \"documents\"" "${STATE_FILE}"
yq -i ".resources.storage.soft_delete_days = 7" "${STATE_FILE}"

yq -i ".resources.app_insights.name = \"${APP_INSIGHTS_NAME}\"" "${STATE_FILE}"
yq -i ".resources.app_insights.id = \"${APPI_RESOURCE_ID}\"" "${STATE_FILE}"
yq -i ".resources.app_insights.connection_string_secret_uri = \"<will-be-stored-in-keyvault-step-03>\"" "${STATE_FILE}"

# 標記 STEP-02 完成
yq -i ".steps_completed += [{\"step_id\": \"STEP-02\", \"completed_at\": \"${NOW}\", \"status\": \"success\"}]" "${STATE_FILE}"
yq -i ".next_step = \"STEP-03\"" "${STATE_FILE}"
```

### Verify
```bash
# 確認 state file 結構正確（YAML 可解析）
yq '.' "${STATE_FILE}" >/dev/null && echo "OK: YAML valid"

# 確認所有資源都已寫入
yq '.resources | keys' "${STATE_FILE}"

# 確認 next_step 已更新
yq '.next_step' "${STATE_FILE}"

# 確認 STEP-02 在 steps_completed 中
yq '.steps_completed[] | select(.step_id == "STEP-02")' "${STATE_FILE}"

# 確認 secrets 未寫入（grep 不該找到任何敏感字串）
! grep -iE 'password|secret|api.?key|connection.?string=' "${STATE_FILE}" | grep -v 'secret_uri' && echo "OK: no secrets leaked"
```

### Expected Output
```

- acr
- container_apps_env
- postgres
- storage

"STEP-03"
{"step_id": "STEP-02", "completed_at": "2026-...", "status": "success"}
```

### If Fails
- Error: `yq: command not found` → STEP-01 Action 1.1 應已驗證，重新安裝 yq
- Error: YAML 結構損壞 → 從 git 還原（state file 不入 git，需從備份還原或重跑 STEP-01 Action 1.8 重建）
- 偵測到 secret 字串 → **立即停止**，刪除 state file 後重做（不可繼續 STEP-03）

## 📝 設計筆記與待確認事項

### 已知設計決策

| 項目 | 決策 | 理由 |
|------|------|------|
| ACR sku | Standard | 支援 webhook + 預留 geo-replication（升級到 Premium 即可） |
| ACR admin user | Disabled | 強制走 Managed Identity（安全強化） |
| PG tier | Burstable B2s | UAT 成本優先，Pilot 規模足夠（50-100 張/週） |
| PG HA | Disabled | UAT 不需要 zone-redundant，Prod 才啟用 |
| Storage sku | Standard_LRS | UAT 不需要 GRS，Prod 用 ZRS/GZRS |
| Storage soft-delete | 7 天 | 誤刪保護，與 PG backup 一致 |
| MI 類型 | System-Assigned | 與 Container App 生命週期綁定，簡化管理 |

### 已知風險

1. **PostgreSQL 建立時間不可控**：5-15 min 為常態，最長至 20 min。若 polling 超過 25 分鐘建議升級。
2. **ACR 全球唯一名稱衝突**：`acraidocextractuat` 若被搶用需改名（更新 STEP-00 §5）。
3. **MI RBAC 賦權延遲**：本 STEP 不執行 RBAC 賦權，必須在 STEP-08 完成後執行 `rbac-pending.sh`，否則 Container App 無法 pull image / 讀 KV / 寫 Storage。

### 與其他 STEP 的關聯

- **依賴 STEP-01**: Prerequisites 全部 pass + state file initialized
- **被 STEP-03 依賴**: Key Vault 建立後寫入 PG admin password / App Insights connection string
- **被 STEP-04 依賴**: ACR login server 用於 docker push
- **被 STEP-05 依賴**: PG FQDN 用於 prisma migrate deploy
- **被 STEP-08 依賴**: CAE resource ID 用於 Container App 建立 + RBAC 賦權
