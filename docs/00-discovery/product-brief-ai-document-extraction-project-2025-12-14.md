# Product Brief: ai-document-extraction-project

**Date:** 2025-12-14
**Author:** chris

---

## Executive Summary

本項目旨在建立一套 AI 驅動的文件內容提取與自動分類系統，專門解決 SCM 部門處理 Freight Invoice 的效率問題。系統將處理來自 100+ 種不同格式的發票（45+ 個 Forwarder），自動提取內容並映射到 90 個統一 Header，目標達成 90-95% 的自動化率和準確率。

APAC 地區每年約有 450,000-500,000 張文件需要處理。透過本系統的實施，預計可節省 35,000-40,000 人時/年，大幅提升工作效率並降低人工成本。更重要的是，結構化的數據將為供應商分析、費用優化和異常檢測提供堅實基礎，釋放現有人員投入更高價值的策略性工作。

---

## Core Vision

### Problem Statement

SCM 部門目前面臨嚴峻的文件處理挑戰：每年需處理大量來自不同貨運代理（Forwarder）的發票，這些發票格式各異、術語不統一，導致數據錄入工作繁瑣且容易出錯。

**用戶日常痛點：**
- 面對堆積如山的 PDF 文件，每張格式都不同
- 眼睛盯著螢幕逐項核對，重複性工作消耗精力
- 擔心出錯影響財務報表，心理壓力大
- 專業能力無法發揮，困在低價值的數據錄入工作中

人工處理方式不僅效率低下，更難以確保數據的一致性和準確性，影響後續的財務報表和業務分析。

### Problem Impact

| 影響面向 | 具體痛點 |
|---------|---------|
| **時間成本** | 若每張發票處理需 5 分鐘，50 萬張 = 約 41,667 人時/年 |
| **準確性風險** | 人工處理 100+ 種格式容易產生錯誤和遺漏 |
| **擴展瓶頸** | 業務增長需按比例增加人力，難以規模化 |
| **數據價值損失** | 數據品質不穩定，管理層無法獲取準確洞察 |
| **機會成本** | 專業人員困在低價值工作，無法投入分析策略 |

### Why Existing Solutions Fall Short

傳統的文件處理方式存在以下局限：

| 現有方案 | 局限性 |
|---------|--------|
| **純手動處理** | 效率極低，無法應對每年 450,000+ 張文件的處理量 |
| **簡單 OCR 方案** | 只能提取文字，無法理解語義和處理格式差異 |
| **固定規則引擎** | 需為每種格式建立獨立規則（100×90=9,000條），維護成本極高 |
| **通用 AI 方案** | 缺乏物流發票領域的專業術語知識，準確率不足 |
| **現有 ERP 整合** | 通常只支援標準格式，無法處理多樣化的 Forwarder 發票 |

**核心問題：** 市面上的方案大多是「通用型」，無法處理物流發票的複雜術語和多樣格式。需要的是**深度垂直整合**的領域專家系統。

### Proposed Solution

建立一套智能化的文件處理平台，核心特色包括：

#### 1. 三層映射架構

```
┌─────────────────────────────────────────────────────────┐
│ TIER 1: Universal Mapping（通用層）                      │
│ • 覆蓋 70-80% 常見術語，所有 Forwarder 通用              │
│ • 維護成本：低（只需維護一份）                           │
├─────────────────────────────────────────────────────────┤
│ TIER 2: Forwarder-Specific Override（特定覆蓋層）        │
│ • 只記錄該 Forwarder 與通用規則「不同」的映射            │
│ • 維護成本：中（每個 Forwarder 只需記錄差異）            │
├─────────────────────────────────────────────────────────┤
│ TIER 3: Learned Mapping（AI 學習層）                     │
│ • 從人工校正中自動學習，持續優化                         │
│ • 同一映射被確認 3+ 次，自動升級為正式規則               │
│ • 維護成本：自動（系統自動學習）                         │
├─────────────────────────────────────────────────────────┤
│ FALLBACK: LLM Classification（智能兜底）                 │
│ • 當以上都無法匹配時，使用 GPT-5.2 智能分類               │
│ • 可處理從未見過的新術語                                 │
└─────────────────────────────────────────────────────────┘

成本對比：傳統方案需 9,000 條規則 → 本架構僅需約 800 條
```

#### 2. 智能信心度評分機制

```
┌────────────────────────────────────────────────────────┐
│ 信心度 > 90%  →  ✅ 自動通過，無需人工介入              │
├────────────────────────────────────────────────────────┤
│ 信心度 70-90% →  ⚡ 快速人工確認（一鍵確認/修正）        │
├────────────────────────────────────────────────────────┤
│ 信心度 < 70%  →  🔍 完整人工審核（詳細檢查）             │
└────────────────────────────────────────────────────────┘

關鍵設計：每次人工修正都會反饋到學習系統，持續提升準確率
```

#### 3. 無縫人機協作體驗

- **直覺的審核界面**：清晰展示 AI 判斷結果和信心度
- **透明的決策理由**：用戶可查看 AI 為何做出該分類
- **順暢的修正流程**：一鍵修正並自動學習
- **漸進式信任建立**：從高審核逐步過渡到低審核模式

#### 4. 優雅的降級策略（Graceful Degradation）

- 新 Forwarder 格式：系統標記為「待建檔」，人工處理並建立 Profile
- AI 無法識別時：自動路由到人工隊列，確保零遺漏
- 服務異常時：本地快取確保核心功能可用

#### 5. 企業級整合

- **文件來源**：SharePoint / Outlook / Azure Blob Storage
- **身份認證**：Azure AD (Entra ID) SSO 統一登入
- **工作流引擎**：n8n 處理業務流程自動化

### 預期效益量化

| 指標 | 目標值 | 說明 |
|------|--------|------|
| 年處理量 | 450,000-500,000 張 | APAC 地區全部文件 |
| 自動化率 | 90-95% | 無需人工介入的比例 |
| 準確率 | 90-95% | AI 分類正確的比例 |
| 節省人時 | 35,000-40,000 人時/年 | 基於每張 5 分鐘估算 |
| 人工處理量 | 25,000-50,000 張/年 | 需人工介入的案例 |

**數據附加價值：**
- 供應商費率分析基礎
- 異常費用自動檢測
- 議價談判數據支撐
- 成本優化洞察報告

### Key Differentiators

| 差異化優勢 | 說明 |
|-----------|------|
| **智能三層架構** | 大幅降低 mapping 維護成本（從 9,000 條降至約 800 條） |
| **持續學習閉環** | 每次人工修正自動學習，3 次確認即升級為正式規則，準確率持續提升 |
| **信心度機制** | 智能平衡自動化效率與人工品控，透明可追溯的 AI 決策 |
| **企業深度整合** | 無縫對接現有 Microsoft 生態系統（Azure AD, SharePoint, Outlook） |
| **數據資產化** | 不只是效率工具，更是建立供應商分析和費用優化的數據基礎 |
| **漸進式上線** | Phase 1 聚焦 Top 10 Forwarder（85% 業務量），快速見效並持續優化 |
| **無縫人機協作** | 直覺的審核體驗，漸進式信任建立機制 |
| **擴展潛力** | APAC 成功後可複製到其他地區，建立全球統一平台 |

### 實施策略

**Phase 1 - 快速見效（聚焦 Top 10 Forwarder）**
- 覆蓋 85% 業務量
- 建立核心三層映射架構
- 驗證 AI 準確率和學習機制

**Phase 2 - 完整覆蓋**
- 擴展至全部 45+ Forwarder
- 優化人機協作體驗
- 建立數據分析儀表板

**Phase 3 - 價值延伸**
- 供應商分析報告
- 異常費用自動檢測
- 跨地區擴展準備

### 誠實的風險披露

| 風險項目 | 應對策略 |
|---------|---------|
| 5-10% 仍需人工處理 | 設計高效的審核界面，確保人工處理體驗順暢 |
| 新 Forwarder 需要建檔 | 標準化 Profile 建立流程，AI 輔助加速 |
| AI 誤判風險 | 信心度機制 + 人工審核閘門，確保品質 |
| 系統學習需要時間 | 漸進式上線，逐步建立信任和準確率 |

## Target Users

### Primary Users

#### Persona 1: 數據處理專員 - 「Amy」

**基本資料：**
- **角色**：SCM 數據處理專員
- **地點**：香港辦公室（11 個 APAC 城市之一）
- **團隊規模**：每個城市至少 1 人，共約 11-15 人
- **日處理量**：平均每人每天約 150-200 張發票

**日常工作情境：**
Amy 每天的工作從檢查郵箱開始，收到各個 Forwarder 發來的發票 PDF。她需要：
1. 從郵件下載附件到本地/SharePoint
2. 辨識發票來源和類型（HEX/HIM/CEX/CIM）
3. 打開 PDF，逐行閱讀費用項目
4. 手動輸入到 Excel 表格中
5. 如果是其他城市的發票，需要請求當地同事協助提取明細

**目前使用的工具：**
- Microsoft Excel（數據錄入）
- SharePoint List（文件管理和協作）
- Outlook（接收發票郵件）

**痛點與挫折：**

| 痛點類型 | 具體描述 |
|---------|---------|
| 😫 **眼睛疲勞** | 每天盯著不同格式的 PDF，逐項核對 |
| 😤 **容易出錯** | 手動輸入時打錯數字或選錯分類 |
| 😩 **跨城市協調困難** | 無法強制要求其他城市提供完整明細 |
| 😞 **重複性工作** | 大量時間花在低價值的數據錄入 |
| 😟 **價值感低落** | 專業能力無法發揮，困在機械性工作中 |

**潛在顧慮（需要主動解決）：**
- 🤔 「AI 會不會取代我的工作？」→ 需要強調『賦能而非取代』
- 😰 「AI 出錯怎麼辦？我會不會被怪罪？」→ 需要設計容錯體驗
- 🙄 「又要學新系統，會不會很麻煩？」→ 需要簡化上手流程

**成功願景：**
「如果系統能自動讀取發票內容並填好表格，我只需要快速確認一下就好，這樣我就有時間做更有意義的分析工作了！而且終於不用擔心打錯字了。」

#### Persona 2: 區域 SCM 經理 - 「David」

**基本資料：**
- **角色**：APAC 區域 SCM 經理
- **職責**：監督 11 個城市的物流成本和數據品質
- **彙報對象**：區域總監 / CFO

**日常工作情境：**
David 需要定期審核各城市提交的物流費用報表，確保數據準確並進行成本分析。但現實是：
- 只能看到「大數」（總金額），缺乏費用明細
- 無法有效核對 Forwarder 收費是否合理
- 跨城市數據格式不一致，難以匯總分析

**關注重點：**

| 關注面向 | 具體需求 |
|---------|---------|
| 📊 **準確率** | 報表數據是否可靠？AI 分類是否正確？ |
| ⚡ **效率** | 團隊是否花太多時間在低價值工作？ |
| 💰 **成本** | 能否發現異常收費或優化機會？ |
| 📈 **可視化** | 能否即時看到處理進度和品質指標？ |

**目前的困境：**
- 無法有效監控數據品質（明細不齊全）
- 難以進行深入的供應商分析
- 團隊被數據錄入工作綁住，無法做策略性工作
- 年度審計時難以提供完整追溯資料

**對 AI 自動化的期望與顧慮：**

| 期望 | 顧慮 |
|------|------|
| 更高效的自動化處理 | AI 準確率能否達到要求？ |
| 完整的費用明細可視化 | 過渡期間如何確保數據品質？ |
| 釋放團隊做高價值工作 | 系統出問題時的回退方案？ |
| 供應商分析和議價能力 | 其他城市是否願意配合？ |

**成功願景：**
「我希望看到完整的費用明細，能夠快速發現異常，而且團隊能把時間花在分析和優化上，而不是數據錄入。最重要的是，審計時能夠完整追溯每一筆數據。」

### User Adoption Considerations

#### 變革管理策略

#### 容錯體驗設計原則

1. **AI 出錯時**
   - 不要讓用戶覺得「果然不可靠」
   - 而是「謝謝你幫忙教 AI」

2. **修正流程**
   - 一鍵修正，不需要繁瑣步驟
   - 即時反饋「已學習」

3. **責任歸屬**
   - 人工確認過的數據，責任清晰
   - AI 自動通過的數據，有完整記錄可追溯

### Out of Scope for MVP

| 項目 | 階段 | 原因 |
|------|------|------|
| **完整多語言支援** | MVP 1.5 | 香港試行階段不需要 |
| **第 3 層學習映射** | MVP 1.5+ | 需要研究實現方式 |
| **完整管理員界面** | MVP 1.5 | MVP 先用簡化配置 |
| **進階分析報告** | Phase 2+ | 需要累積足夠數據 |
| **Forwarder 直接上傳** | 長期 | 需要外部配合 |
| **全球其他地區** | 長期 | 先驗證 APAC |
