  🎯 Azure 部署需要處理的核心項目

  1. 容器化 / 建置

  - 撰寫 Dockerfile（multi-stage Next.js production build）
  - Build target：App Service for Containers / Container Apps / AKS

  2. 基礎設施（Azure 資源）

  - Azure Database for PostgreSQL Flexible Server（取代 docker postgres）
  - Azure Blob Storage（取代 Azurite）
  - Azure Container Apps / App Service（取代本地 npm run dev）
  - Azure Key Vault（存放 AUTH_SECRET / JWT_SECRET / ENCRYPTION_KEY / API keys）
  - Azure Container Registry (ACR)（若自建 image）
  - Network：VNet / Private Endpoints（可選但推薦）

  3. Schema 遷移（🔴 Critical）

  目前規劃的 prisma db push --accept-data-loss 在 production 是 極度危險。需要：
  - 產生並驗證正式的 prisma migrate 流程
  - 或評估 prisma db push 在 prod 的風險（本項目 122 models / 10 migrations 的落差是歷史問題）
  - 定義遷移失敗的 rollback 策略

  4. Seed 策略

  - Prod 不應跑完整 seed（例：dev-user-1 不應在 prod 存在）
  - 需要 prod-seed.ts（只建必要的 roles / regions / system-user）
  - 或完全由 DBA 手動執行一次性 seed

  5. 配置注入

  - Azure App Service / Container Apps 環境變數 ↔ .env 映射
  - Key Vault 敏感值引用（如 @Microsoft.KeyVault(SecretUri=...)）
  - FIX-054 的 SYSTEM_USER_ID 在 prod 對應到真實 system user 的 UUID

  6. CI/CD Pipeline

  - GitHub Actions 或 Azure DevOps
  - main push → build → push ACR → deploy
  - Pull Request → preview slot（可選）
  - Database migration step（在部署前或之間）

  7. 部署前自檢

  - 現有 verify-environment.ts 改造為 Azure-aware
  - 例：檢查 NODE_ENV=production、Key Vault 引用正常、Managed Identity 可用

  8. 運維觀測

  - Application Insights 整合
  - Log Analytics
  - Alerts（延用 AlertRule models？）

  9. 安全強化（比本地更嚴格）

  - HTTPS only / TLS 1.2+
  - AUTH_TRUST_HOST 關閉（已 HTTPS）
  - Azure AD 真實 SSO（取代 dev mode）
  - Rate limit 強制啟用（UPSTASH_REDIS_*）

  現況的風險                                                                                                                  
                                                   
  - 本地使用 prisma db push --accept-data-loss（沒有 migration 歷史）
  - prisma/migrations/ 有 10 個舊 migrations，停留在 2025-12-19

  兩方案對比

  ┌───────────────┬─────────────────────────────────┬───────────────────────────────────┐
  │     面向      │        方案 A：重建基線         │       方案 B：繼續 db push        │
  ├───────────────┼─────────────────────────────────┼───────────────────────────────────┤
  │ Prod Rollback │ ✅ 可靠                         │ 🔴 幾乎不可能                     │
  ├───────────────┼─────────────────────────────────┼───────────────────────────────────┤
  │ 變更追蹤      │ ✅ 每次變更都有 SQL diff        │ ❌ 只有 Prisma Schema diff        │
  ├───────────────┼─────────────────────────────────┼───────────────────────────────────┤
  │ 資料損失風險  │ ✅ 低（migration 明確）         │ 🔴 高（db push 可能 drop column） │
  ├───────────────┼─────────────────────────────────┼───────────────────────────────────┤
  │ 合規 / 審計   │ ✅ 可追溯                       │ ❌ 審計人員會質疑                 │
  ├───────────────┼─────────────────────────────────┼───────────────────────────────────┤
  │ 本地開發影響  │ ⚠️  一次性重置 migration history │ ✅ 無影響                         │
  ├───────────────┼─────────────────────────────────┼───────────────────────────────────┤
  │ Effort        │ 2-3 天（含 PoC 驗證）           │ 0 天（但技術債巨大）              │
  └───────────────┴─────────────────────────────────┴───────────────────────────────────┘

  我的強烈建議：方案 A

  理由：
  1. Prod 環境沒 rollback 能力是不可接受的風險（特別是 450K 張發票/年的業務規模）
  2. FIX-054 / CHANGE-054 剛建立了部署可靠性基線，方案 B 會讓這努力功虧一簣
  3. Prisma 官方明確反對 prod 用 db push
  4. 方案 A 的「一次性本地重置」成本可接受 — 本專案仍是單團隊開發

  ⚠️  方案 A 的隱藏風險（你需要知道）

  - 10 個舊 migrations 如何處置？ 三個選項：
    - (a) 歸檔到 prisma/migrations/_archive/，新基線從 0001_initial_baseline 重頭
    - (b) 保留 10 個舊 migrations + migrate resolve 標記為已執行 + 新增 0011_squashed_since_2025_12_19 補差
    - (c) 完全刪除（最乾淨但最激進）
  - 現有本地開發環境需要重新初始化（團隊所有成員 rm -rf prisma/migrations && npx prisma migrate dev）
  - 需要一個 PoC：空 DB + migrate dev --name initial_baseline → diff 比對現有 prod-like schema → 確認無漏

  🙋‍♂️需要你決定

  1. 接受方案 A？ (Y/N)
