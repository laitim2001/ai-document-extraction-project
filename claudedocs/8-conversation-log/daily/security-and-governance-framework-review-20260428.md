  問題 1：IAM — MFA 與帳號鎖定的取捨                                                                                     
                                                                                                                         
  MFA 部分                                                                                                               
                                                   
  完全同意你的判斷。對內系統 + Azure AD SSO 已經達到企業級基準，因為：

  - SSO 本身已將認證責任委派給 Azure AD（IT 部門通常已在 AAD 層面強制 MFA / Conditional Access）
  - 應用層再做一次 MFA 是「重複防禦」，徒增 UX 摩擦
  - 業界實踐：SSO + AAD Conditional Access 等同於應用層 MFA，符合 SOC2 / ISO 27001 要求

  建議調整 IAM-06：
  原: MFA 對 admin 強制 (L3+)
  改: SSO 強制（已達成）；MFA 由 AAD Conditional Access 控管（應用層免實作）

  帳號鎖定部分

  你的觀察非常正確——「只做鎖定不做解鎖」是常見的半成品反模式。

  完整的帳號鎖定功能必須包含 5 個配套：

  ┌─────┬─────────────────────────────────────────┬──────────────────────────────┐
  │  #  │                配套功能                 │          缺了會怎樣          │
  ├─────┼─────────────────────────────────────────┼──────────────────────────────┤
  │ 1   │ 觸發鎖定（N 次失敗）                    │ 基礎防 brute force           │
  ├─────┼─────────────────────────────────────────┼──────────────────────────────┤
  │ 2   │ 自動解鎖（時間衰減，e.g., 15 分鐘）     │ 否則用戶必須打給 IT          │
  ├─────┼─────────────────────────────────────────┼──────────────────────────────┤
  │ 3   │ 管理員手動解鎖介面                      │ IT 需要工具，不能改 DB       │
  ├─────┼─────────────────────────────────────────┼──────────────────────────────┤
  │ 4   │ 用戶自助解鎖（email 驗證碼 / SSO 重置） │ 對內系統可接受，但需有此選項 │
  ├─────┼─────────────────────────────────────────┼──────────────────────────────┤
  │ 5   │ 解鎖審計日誌                            │ 誰在何時解鎖了誰             │
  └─────┴─────────────────────────────────────────┴──────────────────────────────┘

  建議調整 IAM-07：
  原: 帳號鎖定 (單一檢查項)
  改: 拆分為 IAM-07a (鎖定觸發) + IAM-07b (解鎖流程) + IAM-07c (鎖定審計)
     ⚠️  三項必須同時達 L2+，否則此項評為 L0（半成品比沒做還危險）

  對內系統的務實建議：因為已有 SSO，本地帳號使用率應該很低。可以只實作 SSO

  B9 — HSTS preload 是什麼？

  HSTS（HTTP Strict Transport Security）基本功能：
  HTTP response header 告訴瀏覽器「這個網站永遠用 HTTPS，不要用 HTTP」。範例：
  Strict-Transport-Security: max-age=31536000; includeSubDomains

  HSTS preload 的差異：

  ┌──────────┬───────────────────────────────────────────────────────────────────┬───────────────────────────────┐
  │   機制   │                               說明                                │             風險              │
  ├──────────┼───────────────────────────────────────────────────────────────────┼───────────────────────────────┤
  │ 基本     │ 用戶第一次訪問時瀏覽器才知道規則。第一次訪問可能仍走 HTTP（被     │ 低風險                        │
  │ HSTS     │ MITM 攻擊風險）                                                   │                               │
  ├──────────┼───────────────────────────────────────────────────────────────────┼───────────────────────────────┤
  │ HSTS     │ 把網域提交到 https://hstspreload.org，Chrome/Firefox/Safari       │ 不可逆：移除網域要 6-12       │
  │ preload  │ 內建清單，連第一次訪問都強制 HTTPS                                │ 個月才從瀏覽器清單清掉        │
  └──────────┴───────────────────────────────────────────────────────────────────┴───────────────────────────────┘

  為何說「不可逆」是風險：
  - 若提交了 preload 後，未來某天 prod HTTPS 證書過期/配置錯誤 → 用戶完全無法訪問（瀏覽器拒絕 fallback 到 HTTP）
  - 即使你立即修復 HTTPS，舊用戶的瀏覽器快取中仍有 preload 規則
  - 移除流程：提交移除請求 → 等待 6-12 個月隨瀏覽器版本更新清除

  1. Microsoft 企業折扣未計入：實際 Microsoft EA / CSP 合約通常有 5-15% 折扣，實際支付可能比估算便宜
