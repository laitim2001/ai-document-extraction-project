  問題 2：CLAUDE.md ↔ codebase-analyze 自動對照機制

  ✅ 技術完全可行，以下是三種可行方案，從輕到重：

  方案 A：Pre-commit Hook（最輕量，建議起步）

  在 .claude/settings.json 加一個 PreCompact 或 Stop hook，每次 session 結束或 commit 前執行 bash 腳本：

  # scripts/verify-claude-md-sync.sh
  # 1. 從 codebase 實測統計（服務數、API 數、組件數、i18n、Prisma models）
  # 2. 從 CLAUDE.md 文件提取聲稱數字
  # 3. 若差異 > 5%，echo 警告到 stderr

  優點：零額外成本、可直接透過 Claude Code hook 觸發
  缺點：只檢查數字，不檢查架構/安全發現

  方案 B：季度性 Refresh Skill（中等投入，推薦）

  建立 /refresh-codebase-analysis 自訂 Skill，定期（例如 Sprint 結束或每季）執行：

  1. 重新掃瞄 codebase → 產生新的 00-analysis-index.md 統計
  2. Diff 舊版 vs 新版 → 列出變化超過閾值的項目
  3. 自動在 CLAUDE.md 寫入「⚠️  待人工確認差異」區塊
  4. 產生 CHANGE-XXX-claude-md-sync.md 規劃文件

  優點：保持 codebase-analyze 活躍、結構化產出
  缺點：需要寫 skill（1-2 小時）

  方案 C：CI Job（最完整）

  在 GitHub Actions 加一個 verify-docs-consistency.yml，PR 時自動比對。

  優點：PR 合入前強制檢查
  缺點：需要 CI 配置、回饋循環較長

  我的建議組合：A + B

  - 立即：做方案 A（腳本 + hook），避免再次漂移
  - 每季：用方案 B 做深度 refresh，產生新的 verification 報告（模仿現有 R1-R15 格式）
  - 不做：方案 C（專案是單人/小團隊，CI 成本高於收益）

  → 結論：可行，codebase-analyze 的投資不會浪費。建議先做方案 A 確保「不再惡化」，之後再討論方案 B 的時程。

  ❓ 為什麼會有「升級為 CHANGE」這個選項

  這涉及 FIX vs CHANGE 的定性差異：

  ┌────────┬───────────────────────────────────┬──────────────────────┐
  │  類型  │               定義                │       適用場景       │
  ├────────┼───────────────────────────────────┼──────────────────────┤
  │ FIX    │ 修復 Bug — 恢復原本應有的正確行為 │ 代碼行為偏離設計規格 │
  ├────────┼───────────────────────────────────┼──────────────────────┤
  │ CHANGE │ 功能變更 — 主動改變業務行為       │ 需要產品/業務方決策  │
  └────────┴───────────────────────────────────┴──────────────────────┘

  這個議題卡在灰色地帶，原因有三個：

  1. 「哪個行為正確」不是純技術問題

  - generateRoutingDecision（降級策略）說：新公司 + 高信心度 → QUICK_REVIEW 就夠了
  - getSmartReviewType（強制策略）說：新公司 → 必須 FULL_REVIEW 以建立完整識別規則

  這兩種策略在業務上都合理：
  - 策略 A（降級）優先使用者體驗，減少審核負擔
  - 策略 B（強制）優先資料品質，確保新公司的識別規則一次到位

  技術面無從判斷哪個是 Bug、哪個是正確設計——這取決於業務方對「新公司首次處理應承擔多少人工成本」的決策。

  2. 修復後會實質改變生產行為

  雖然 getSmartReviewType 在生產代碼零調用，但 generateRoutingDecision 的邏輯本身沒變。然而：
  - 若統一後認定策略 A 正確，未來不會有人誤用強制策略（維持現況）
  - 若統一後認定策略 B 正確，需要反向修改 generateRoutingDecision，這會實質改變 pipeline
  路由結果——例如原本信心度 98% 的新公司文件會自動通過，改後會強制人工審核。這對既有使用者是可感知的功能變更。

  如果修復方向是策略 B，本質上就是 CHANGE 而非 FIX（主動改變業務邏輯）。

  3. CHANGE 需要不同的驗收流程

  - FIX 流程：確認 Bug → 修復 → code review → 合入
  - CHANGE 流程：業務需求確認 → PRD/Tech Spec → 產品方 sign-off → 實作 → UAT

  如果這個問題實質上是「業務決策」而非「代碼錯誤」，強行走 FIX 流程會跳過業務 sign-off
  這一步，導致後續爭議（業務方可能說「我從沒同意過新公司要降級」）。

  Upstash Redis 簡介

  Upstash 是一家提供雲端託管 Redis服務的公司，特別適合 Serverless / Next.js 應用：

  ┌────────────────────────────────────┬────────────────────────────────────────────────────────────┐
  │             傳統 Redis             │                       Upstash Redis                        │
  ├────────────────────────────────────┼────────────────────────────────────────────────────────────┤
  │ 需自架伺服器/容器                  │ 全託管（無伺服器維護）                                     │
  ├────────────────────────────────────┼────────────────────────────────────────────────────────────┤
  │ 用 TCP 連線（Edge Runtime 不支援） │ 用 HTTPS REST API（Edge/Vercel/Cloudflare Workers 皆支援） │
  ├────────────────────────────────────┼────────────────────────────────────────────────────────────┤
  │ 按實例計費（24/7 都付費）          │ 按請求數計費（無請求不付錢）                               │
  ├────────────────────────────────────┼────────────────────────────────────────────────────────────┤
  │ 跨網路延遲高                       │ 多區域部署，edge location 低延遲                           │
  └────────────────────────────────────┴────────────────────────────────────────────────────────────┘

  本專案為什麼需要 Upstash？

  看 package.json 已安裝 @upstash/redis@^1.35.8，但 src/ 內零
  import，代表專案原本就規劃使用但一直沒實作。FIX-052 就是把它真正接上。

  不配置 Upstash 會怎樣？

  現況（無 Upstash）：
  - ✅ 本地開發正常：service 自動 fallback 為 in-memory Map，首次呼叫時 log 一次 [RateLimit] UPSTASH_REDIS_*
  not configured WARN
  - ✅ 單實例生產部署正常：in-memory 足夠
  - ❌ 多實例部署失效：如 PM2 cluster、Vercel 多函數、Kubernetes HPA、Azure App Service scale-out ——
  每個實例有獨立計數，rate limit 會失效

  配置 Upstash 後的影響

  好處：
  - 🟢 Rate limit 跨實例同步（多實例計數共用）
  - 🟢 服務重啟不遺失 rate limit 記錄（Redis 持久化）
  - 🟢 可支援 Edge Runtime（Vercel Edge Functions 等）

  代價：
  - 💰 免費方案：10,000 commands/day（一般開發量夠用）
  - 💰 付費方案：按用量計費（低流量通常 < $10/月）
  - ⚙️  配置 2 個環境變數：UPSTASH_REDIS_REST_URL、UPSTASH_REDIS_REST_TOKEN（從 Upstash Console 取得）

  短期建議

  目前不用配置。專案若是單實例部署（如 Azure App Service 單實例、或本地開發），現有 in-memory fallback
  已足夠。只有當你決定多實例 scale-out 時才需要實際配置 Upstash。
